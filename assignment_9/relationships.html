TYPES OF RELATIONSHIPS IN DBMS

ONE TO ONE (1:1) RELATIONSHIP

In a one-to-one relationship, a single record in one table is associated with a single record in another table. This type of relationship is often used when you want to split data into two tables but still maintain a direct link between them.

  1. Example: Each person has one unique passport.
  2. Implementation: Create two tables, `Person` and `Passport`, where the primary key of `Person` is also a foreign key in `Passport`.

ONE TO MANY (1:M) RELATIONSHIP

In a one-to-many relationship, a single record in one table is associated with multiple records in another table. This is one of the most common types of relationships in databases.

  1. Example: A single customer can have multiple orders.
  2. Implementation: Create two tables, `Customer` and `Order`, where the primary key of `Customer` is a foreign key in `Order`.

MANY TO MANY (M:M) RELATIONSHIP

In a many-to-many relationship, multiple records in one table are associated with multiple records in another table. This type of relationship is typically implemented using a junction table.

  1. Example: Students can enroll in multiple courses, and each course can have multiple students.
  2. Implementation: Create three tables, `Student`, `Course`, and a junction table `Enrollment` that contains foreign keys referencing both `Student` and `Course`.

MANY TO ONE (M:1) RELATIONSHIP

In a many-to-one relationship, multiple records in one table are associated with a single record in another table. This is essentially the reverse of a one-to-many relationship.

  1. Example: Multiple employees can work in a single department.
  2. Implementation: Create two tables, `Employee` and `Department`, where the primary key of `Department` is a foreign key in `Employee`.

SELF REFERENCING (1:1) RELATIONSHIP

In a self-referencing relationship, a table contains a relationship with itself. This is used to model hierarchical data or recursive relationships.

  1. Example: An employee can be a manager of other employees.
  2. Implementation: Create a single table `Employee` with a foreign key that references the primary key of the same table to indicate the manager-employee relationship.
